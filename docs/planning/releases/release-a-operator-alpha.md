# Release A: Operator Alpha

## 릴리즈 미션

Jarvis가 Codex의 작업 단계를 `관찰 -> 판단 후보 생성 -> 다음 단계 전송` 수준까지 안정적으로 수행하게 만드는 첫 번째 실전 릴리즈입니다.

## 해결하려는 문제

현재 상단장님이 직접 하고 있는 반복 업무는 아래와 같습니다.

- Codex가 작업을 마쳤는지 수시로 확인
- 결과 화면이 크게 이상 없는지 확인
- 다음 단계 진행 요청
- 문제가 있으면 멈추고 다시 지시

Release A는 이 중에서도 가장 반복적이고 시간이 많이 드는 운영 업무를 먼저 대체합니다.

## 핵심 사용자 시나리오

1. 상단장님이 프로젝트 목표와 단계 목록을 입력합니다.
2. Jarvis가 Codex 창을 안정적으로 붙잡습니다.
3. Jarvis가 화면을 주기적으로 캡처하고 안정화 여부를 판단합니다.
4. 상단장님이 필요 시 프롬프트를 검토하거나, 자동 루프를 켭니다.
5. Jarvis가 다음 단계 프롬프트를 넣고 작업 이력을 남깁니다.
6. 상단장님은 중간중간 로그와 캡처만 확인합니다.

## 이번 릴리즈의 성공 기준

### 기능 기준

- Codex 대상 창을 안정적으로 식별할 수 있음
- 입력창 캘리브레이션 후 프롬프트 전송 정확도가 높음
- 단계 포인터와 프롬프트 조합이 꼬이지 않음
- 자동 루프가 최소한의 안전장치와 함께 동작함
- 모든 액션과 캡처가 추적 가능하게 남음

### 운영 기준

- 한 개 프로젝트에서 최소 1시간 이상 연속 운영 가능
- 상단장님 개입 빈도가 현재보다 체감상 크게 줄어듦
- “왜 이런 행동을 했는지” 로그로 설명 가능

### 품질 기준

- 실수로 엉뚱한 창에 붙여넣지 않음
- 무한 반복 전송을 방지함
- 긴급 정지가 즉시 작동함

## 범위 포함

- Codex 창 탐색 및 타겟 고정
- 입력창 위치 캘리브레이션
- 단계 큐와 프롬프트 생성
- 세션 저장/복원
- 화면 캡처와 증거 저장
- 안정화 감지 고도화
- 운영 대시보드와 액션 타임라인
- 안전 모드와 긴급 정지
- 릴리즈 스모크 테스트 체계

## 범위 제외

- OCR 기반 본문 판독
- 브라우저 결과 화면 의미 이해
- 자동 QA 합격/불합격 판정
- 음성 명령과 웨이크워드
- 외부 알림, 모바일 연동, 원격 승인

## 제품 원칙

- 자동화는 항상 사람이 끊을 수 있어야 합니다.
- 상태는 행동보다 먼저 보여야 합니다.
- 전송은 빠르되, 잘못된 전송 가능성은 최대한 낮춰야 합니다.
- 상단장님이 “무슨 일이 벌어지고 있는지” 항상 이해할 수 있어야 합니다.

## 핵심 지표

### 운영 지표

- Codex 창 타겟 성공률
- 전송 성공률
- 자동 루프 연속 성공 사이클 수
- 긴급 정지 반응 시간

### 신뢰 지표

- 잘못된 창 전송 건수
- 오탐 안정화 건수
- 수동 개입 발생 사유 분포

### 경험 지표

- 프로젝트 설정 완료 시간
- 다음 단계 수동 전송까지 걸리는 시간
- 로그에서 원인 파악에 걸리는 시간

## 아키텍처 영향

Release A에서 반드시 인터페이스를 분리해야 하는 영역은 아래와 같습니다.

- 창 제어 레이어
- 액션 실행 레이어
- 관찰 레이어
- 상태 저장 레이어
- 오케스트레이션 레이어

이렇게 나눠야 Release B에서 OCR/비전이 들어와도 기존 구조를 크게 바꾸지 않습니다.

## 릴리즈 게이트

정식으로 Release A 완료 선언을 하려면 아래를 모두 만족해야 합니다.

1. 실전 프로젝트 3회 이상 운영
2. 긴급 정지, 수동 전송, 자동 루프가 모두 정상 동작
3. 액션 로그와 캡처 근거를 통해 이슈 재현 가능
4. 입력창 캘리브레이션 재설정이 쉬움
5. false positive로 인한 무의미한 다음 단계 전송이 허용 수준 이내

## 다음 릴리즈로 넘겨야 할 것

Release A가 끝나면 바로 Release B에서 다뤄야 할 우선 과제는 아래입니다.

- Codex 본문 OCR
- Codex 상태 분류
- 브라우저 결과 읽기
- 자동 QA 분기
