# Sprint 1: Control Foundation

## 스프린트 목표

javis가 Codex를 안정적으로 `찾고`, `붙잡고`, `입력할 수 있는 상태`를 먼저 완성합니다.

## 왜 먼저 하는가

Release A에서 가장 치명적인 실패는 엉뚱한 창을 조작하거나, 입력창 위치가 어긋나거나, 세션이 날아가는 것입니다. Sprint 1은 이 바닥을 고정하는 단계입니다.

## 포함 티켓

- `A-001` 창 타겟 하드닝
- `A-002` 입력창 캘리브레이션
- `A-004` 세션 저장/복원 v1

## 상세 목표

### 1) 창 제어 신뢰도 확보

- Codex 창이 여러 개 보이거나 제목이 일부 바뀌어도 타겟을 안정적으로 고를 수 있어야 합니다.
- 마지막으로 성공한 창을 다시 붙잡는 흐름이 필요합니다.

### 2) 입력창 캘리브레이션 가능

- 상단장님이 실제 Codex 입력창 위치를 쉽게 잡고 테스트할 수 있어야 합니다.
- 캘리브레이션 결과가 세션에 저장되어 재사용돼야 합니다.

### 3) 세션 유실 방지

- 프로젝트 요약, 목표, 단계, 운영 규칙, 포인터 정보가 저장되어야 합니다.
- 앱을 껐다 켜도 이어서 작업할 수 있어야 합니다.

## 완료 조건

- Codex 창을 붙잡는 성공률이 눈에 띄게 안정적임
- 수동 테스트 전송이 재현 가능하게 성공함
- 세션 저장 후 재실행해도 상태가 유지됨

## 데모 시나리오

1. javis 실행
2. 프로젝트/단계 입력
3. Codex 창 식별
4. 캘리브레이션 모드 진입
5. 테스트 전송
6. 앱 종료 후 재실행
7. 동일 세션 복원 확인

## 주요 리스크

- Codex 창 탐색 기준이 너무 단순할 수 있음
- 상대 좌표가 해상도/창 크기에 민감할 수 있음
- 저장 스키마 변경이 잦으면 이후 하위 기능이 흔들릴 수 있음
