# Native vs Fallback Matrix v1

## 목적

`javis`가 다시 클릭 매크로 중심 앱으로 돌아가지 않도록, `Codex 네이티브 우선 / 로컬 fallback 보조` 원칙을 문서와 제품 양쪽에서 고정합니다.

## 핵심 원칙

### 1. 네이티브 우선

먼저 아래를 확인합니다.

- Codex app 기본 흐름으로 충분한가
- 같은 스레드 운영으로 충분한가
- thread automation 또는 project automation으로 충분한가
- worktree, review queue, skills 조합으로 충분한가

위 조건으로 해결되면 `javis`는 직접 실행보다 감독과 요약에 집중합니다.

### 2. fallback은 보조 수단

로컬 데스크톱 제어 fallback은 아래일 때만 고려합니다.

- Codex 네이티브 경로만으로는 외부 상태 확인이 부족함
- 화면, 브라우저, 팝업 상호작용처럼 데스크톱 표면 보조가 꼭 필요함
- 운영자 경험상 `javis`가 즉시 신호를 보여줘야 함

### 3. 애매하면 진행보다 보류

아래는 자동 진행보다 `pause / ask_user`가 우선입니다.

- 파괴적이거나 되돌리기 어려운 작업
- 배포, 삭제, 과금처럼 운영 리스크가 큰 작업
- 같은 실패가 반복되는 작업
- 확신이 낮고 추정이 많은 상태

## 매트릭스

| 상황 | 기본 선택 | 이유 |
| --- | --- | --- |
| 단계 순차 진행 | Codex 스레드 운영 | 같은 문맥 유지가 쉽고 별도 앱 로직이 적음 |
| 주기적 follow-up | thread automation | 같은 스레드를 다시 깨우는 데 적합 |
| 독립적인 반복 작업 | project automation | 실행 결과를 분리해서 보기 좋음 |
| 코드 변경 가능 반복 작업 | project automation + worktree | 로컬 작업본 오염 위험을 줄임 |
| 화면/브라우저 감시 필요 | javis visual / desktop fallback | Codex 네이티브만으로는 부족할 수 있음 |
| 위험도 높음 / 애매함 | ask_user | 잘못 자동화할 비용이 큼 |

## Release 2 적용 기준

- Codex 전략 센터는 먼저 `추천 네이티브 경로`를 보여줘야 합니다.
- 런북은 `바로 로컬 제어`보다 `Codex에서 어떻게 운영할지`를 먼저 안내해야 합니다.
- UI 문구에서도 `fallback`을 보조 수단으로만 설명해야 합니다.

## 한 줄 결론

`javis`는 먼저 Codex의 손발을 빌리고, 로컬 데스크톱 제어는 정말 필요한 빈틈만 메우는 방향으로 갑니다.
