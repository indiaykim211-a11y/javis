# Phase 2: Codex-First Foundation

## 단계 목적

이 단계의 목적은 `javis가 Codex를 다시 만드는 실수`를 피하는 것입니다.

핵심 질문은 하나입니다.

`이 기능을 우리가 새로 만들어야 하는가, 아니면 Codex 앱이 이미 잘하는가?`

이 단계에서는 최대한 많은 실행 책임을 Codex 네이티브 기능으로 넘기고, `javis`는 그 위를 감독하는 구조로 정렬합니다.

## 왜 이 단계가 먼저 필요한가

`javis`를 너무 빨리 키우면 아래 문제가 생깁니다.

- Codex가 이미 잘하는 기능을 중복 구현하게 됨
- 데스크톱 클릭 자동화에 너무 빨리 의존하게 됨
- 유지보수 포인트가 불필요하게 늘어남
- 나중에 Codex 앱 기능이 좋아질수록 javis 코드가 짐이 될 수 있음

그래서 초반에는 `무엇을 만들까`보다 `무엇을 안 만들까`를 먼저 정하는 것이 맞습니다.

## Codex 네이티브 기능 맵

공식 문서 기준으로 지금 우선 활용해야 하는 기능은 아래와 같습니다.

### 1. Cloud task 실행

- Codex는 클라우드 환경에서 백그라운드 작업을 돌릴 수 있습니다.
- 병렬 작업도 가능합니다.
- 장시간 실행을 로컬 클릭 매크로로 흉내 내는 것보다 훨씬 자연스러운 기본 실행층입니다.

### 2. Multi-agent / project 흐름

- 프로젝트별로 여러 작업을 병렬로 돌릴 수 있습니다.
- 같은 저장소에서도 worktree를 통해 충돌을 줄일 수 있습니다.

### 3. Review queue / Triage

- 결과가 있으면 Triage로 올라오고, 다시 검토를 이어갈 수 있습니다.
- 즉 `작업 결과를 한 번 모아 보는 자리`가 이미 있습니다.

### 4. Automations

- 일정에 따라 백그라운드에서 반복 작업을 돌릴 수 있습니다.
- thread automation과 standalone / project automation을 나눠 쓸 수 있습니다.

### 5. Skills / Plugins / Rules

- 자주 반복되는 운영 방식을 skill로 묶을 수 있습니다.
- plugins와 rules를 조합해 더 안정적인 자동화를 만들 수 있습니다.

## 이 단계에서 javis가 직접 하지 않을 것

적어도 초반에는 아래 기능을 `javis 1차 책임`으로 두지 않습니다.

- 자체 장기 작업 스케줄러를 새로 만들기
- 자체 병렬 작업 엔진 만들기
- 자체 worktree 관리자 만들기
- 자체 review queue 만들기
- Codex 앱을 계속 클릭해서 같은 일을 흉내 내는 것

즉 `Codex가 이미 할 수 있는 실행 기능`은 최대한 그대로 씁니다.

## 이 단계에서 javis가 맡을 것

`javis`는 실행 엔진보다 `감독 레이어`에 집중합니다.

### 1. 마스터플랜 로드

- 상단장님이 확정한 개발 마스터플랜을 읽고
- Codex가 어느 단계에 있는지 연결합니다.

### 2. 정책 적용

- 진행 기준
- 보류 기준
- 재지시 기준
- 사용자 보고 톤
- 안전 규칙

이것을 일관되게 유지합니다.

### 3. 상태 통합

- Codex 상태
- 최근 결과
- 필요 시 화면 증거
- 사용자 승인 여부

를 한 번에 모아서 보여줍니다.

### 4. 상단장님용 Surface

- 작은 팝업으로 지금 상태를 짧게 보여주기
- 왜 멈췄는지, 왜 진행 가능한지 설명하기
- 필요할 때만 Control Center로 보내기

### 5. 나중에 붙을 보강 계층 준비

- 화면 인식
- 음성
- App Server 같은 더 직접적인 연동
- 데스크톱 fallback 제어

## 책임 분리 원칙

### Codex가 맡는 것

- 실제 코드 작성 / 수정
- 테스트 실행
- 병렬 작업
- background run
- worktree 기반 격리
- 반복 작업의 scheduled run

### javis가 맡는 것

- 상단장님 목표와 정책을 구조화
- 마스터플랜 기준 단계 포인터 관리
- Codex 결과를 제품 운영 관점으로 해석
- 상단장님께 짧게 보고
- 필요 시 다음 행동을 제안

### OpenAI 판단 엔진이 맡는 것

- continue / pause / retry / ask-user 결정
- 재지시 문구 생성
- 위험도와 확신도 산출

## 이 단계의 기본 판단 원칙

### 원칙 1. 먼저 Codex 네이티브 경로를 찾는다

새 기능 요청이 나오면 먼저 아래를 확인합니다.

- Codex app에서 이미 가능한가
- cloud task로 충분한가
- automation으로 충분한가
- skill로 패키징하면 되는가

### 원칙 2. 로컬 데스크톱 제어는 fallback이다

클릭 자동화나 화면 매크로는 아래일 때만 고려합니다.

- Codex 네이티브 경로가 없음
- 화면 읽기 / 브라우저 확인처럼 별도 감시가 꼭 필요함
- 사용자 경험상 팝업이 직접 보조해야 함

### 원칙 3. 항상 실행보다 항상 감독을 먼저 만든다

초기 javis는 `항상 모든 걸 직접 수행하는 앱`보다
`Codex가 한 일을 계속 감독하는 앱`에 가깝게 가져갑니다.

## 검증해야 할 초기 시나리오

이 단계에서 먼저 검증할 시나리오는 아래입니다.

### 시나리오 A. 마스터플랜 기반 장기 개발

- 상단장님이 마스터플랜 확정
- Codex가 그 플랜에 따라 장기 작업 수행
- javis는 현재 단계와 다음 액션만 감독

### 시나리오 B. Automations로 반복 추적

- 정해진 스레드나 프로젝트를 주기적으로 다시 확인
- 결과가 있으면 Triage 또는 팝업에 반영

### 시나리오 C. review queue 중심 운영

- 결과가 올라오면 상단장님은 큰 콘솔 대신 요약만 봄
- 세부 검토는 필요할 때만 열기

## 완료 조건

- `Codex가 이미 잘하는 것`과 `javis가 새로 해야 할 것`이 명확히 분리됨
- Release 2 이후 계획이 `Codex-first` 방향으로 흔들리지 않음
- 데스크톱 클릭 자동화가 주 경로가 아니라 fallback 경로로 정리됨
- Automations 활용 전략으로 자연스럽게 이어질 수 있음
