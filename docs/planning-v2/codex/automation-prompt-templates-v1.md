# Automation Prompt Templates v1

## 문서 목적

이 문서는 Codex Automations를 만들 때 바로 가져다 쓸 수 있는 `프롬프트 뼈대`를 정리한 문서입니다.

원칙은 아래와 같습니다.

- 길어도 괜찮지만 애매하면 안 됨
- 무엇을 보고, 무엇을 하지 말고, 언제 멈출지 적어야 함
- 상단장님이 원하는 운영 방식을 durable하게 써야 함

## 템플릿 1. 마스터플랜 Follow-up Heartbeat

### 추천 타입

- `thread automation`

### 템플릿

```text
현재 스레드의 최근 진행 상황을 다시 확인해 주세요.

목표:
- 이 개발 플랜이 지금 어디까지 왔는지 파악
- 다음 단계로 넘어가도 되는지 판단
- 막힘, 반복 실패, 애매한 상태가 있으면 바로 보류

작업 방식:
- 최근 응답과 최근 결과를 먼저 읽고 현재 단계 위치를 추정합니다.
- 지금 바로 진행 가능한지, 조금 더 기다려야 하는지, 사람 확인이 필요한지 구분합니다.
- 진행 가능하면 다음 액션을 짧게 제안합니다.
- 애매하면 진행하지 말고 왜 보류했는지 짧게 설명합니다.

보고 형식:
- 현재 상태
- 판단 이유
- 추천 다음 행동
- 사람 확인 필요 여부
```

## 템플릿 2. Release Smoke Runner

### 추천 타입

- `project automation`

### 템플릿

```text
현재 프로젝트에서 release smoke를 실행해 주세요.

목표:
- 릴리즈 핵심 흐름이 아직 살아 있는지 빠르게 확인

작업 방식:
- 문서화된 smoke runner가 있으면 먼저 실행합니다.
- 실패 시 가장 중요한 실패 지점 1~3개만 요약합니다.
- 성공 시 통과 범위를 짧게 정리합니다.
- 불필요한 대규모 수정은 하지 않습니다.

보고 형식:
- 통과 / 보류 / 실패
- 핵심 점검 범위
- 실패 시 가장 중요한 원인 후보
- 다음 권장 행동
```

## 템플릿 3. PR Babysitter

### 추천 타입

- `thread automation` 또는 `project automation`

### 템플릿

```text
현재 PR 또는 관련 작업 흐름을 다시 확인해 주세요.

목표:
- 새 리뷰 코멘트, requested change, merge blocker가 생겼는지 확인

작업 방식:
- 최근 리뷰 상태를 확인합니다.
- 새 액션이 없으면 짧게 상태만 남깁니다.
- 수정이 필요하면 어떤 피드백이 핵심인지 짧게 정리합니다.
- 자동 수정은 확실한 범위일 때만 제안합니다.

보고 형식:
- 현재 PR 상태
- 새 변화 유무
- 핵심 blocker
- 다음 권장 행동
```

## 템플릿 4. Nightly Project Brief

### 추천 타입

- `standalone / project automation`

### 템플릿

```text
최근 프로젝트 변화를 짧은 브리핑으로 정리해 주세요.

목표:
- 상단장님이 아침에 1분 안에 상황을 파악하게 하기

작업 방식:
- 최근 변경, 진행 상황, 주요 이슈를 workstream 단위로 묶습니다.
- 커밋 나열보다 의미 중심으로 요약합니다.
- 중요 변화가 없는 영역은 길게 쓰지 않습니다.

보고 형식:
- 오늘의 한 줄 상태
- 핵심 workstream 1~3개
- 주의할 위험 1~2개
- 오늘 먼저 볼 것
```

## 템플릿 5. CI Failure Triage

### 추천 타입

- `project automation`

### 템플릿

```text
현재 프로젝트의 CI 실패를 빠르게 triage해 주세요.

목표:
- 실패 원인을 상단장님이 바로 이해할 수 있는 수준으로 줄이기

작업 방식:
- 실패 로그를 확인합니다.
- 실패 유형을 분류합니다: build / test / lint / env / flaky
- 가장 가능성 높은 원인 후보를 1~3개만 남깁니다.
- 무관한 추측은 늘리지 않습니다.

보고 형식:
- 실패 유형
- 가장 가능성 높은 원인
- 관련 파일 또는 영역
- 다음 권장 행동
```

## 템플릿 6. Recent Code Self-Bugfix

### 추천 타입

- `project automation + skill`

### 템플릿

```text
최근 내 변경에서 생긴 문제 후보를 찾아 주세요.

목표:
- 최근 변경과 직접 연결된 버그만 찾고, 가능하면 좁은 범위에서 수정

작업 방식:
- 최근 변경 범위를 먼저 한정합니다.
- 그 범위와 직접 연결된 실패만 찾습니다.
- 관련 없는 오래된 문제는 건너뜁니다.
- 수정할 경우에도 좁고 검증 가능한 범위만 다룹니다.

보고 형식:
- 버그 후보 유무
- 최근 변경과의 연결 근거
- 수정 가능 여부
- 다음 권장 행동
```

## 프롬프트 작성 규칙

### 1. 목적부터 쓴다

- 이 자동화가 왜 존재하는지 먼저 적습니다.

### 2. 작업 방식을 제한한다

- 무엇을 먼저 보고
- 무엇을 하지 말고
- 어디서 멈출지 적습니다.

### 3. 보고 형식을 고정한다

- 상태
- 이유
- 위험
- 다음 행동

이 4개 구조를 기본으로 가져가는 것이 좋습니다.

### 4. 공격적 자동 실행은 초반에 피한다

- 자동 병합
- 무제한 수정
- 광범위한 리팩토링

이런 건 초반에는 넣지 않는 것이 안전합니다.

## javis 연결 메모

나중에 `javis`는 이 템플릿들을 그대로 보여주는 것이 아니라 아래처럼 감싸서 쓸 수 있습니다.

- Control Center에서 템플릿 선택
- 프로젝트 정책과 결합
- 상단장님 취향에 맞는 보고 톤 덧씌우기
- thread / project automation 생성기로 넘기기

## 한 줄 결론

좋은 automation prompt는 `똑똑한 장문`이 아니라 `목적, 제한, 멈춤 기준, 보고 형식이 분명한 지시문`입니다.
