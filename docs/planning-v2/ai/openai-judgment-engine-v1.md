# OpenAI Judgment Engine v1

## 문서 목적

이 문서는 `javis` 안에서 OpenAI가 어떤 역할을 맡고, 어떤 입력을 받아, 어떤 판단을 반환해야 하는지 정의합니다.

핵심 목적은 아래 두 가지입니다.

- OpenAI를 단순 채팅 API가 아니라 `판단 엔진`으로 사용하기
- 자동화가 흔들리지 않도록 입력과 출력 구조를 고정하기

세부 계약은 아래 문서를 같이 봅니다.

- 입력 계약: `../specs/judgment-input-contract-v1.md`
- 출력 계약: `../specs/judgment-response-contract-v1.md`

## 엔진 한 줄 정의

`OpenAI Judgment Engine`은 Codex 진행 상황, 현재 단계, 운영 정책, 나중에는 화면 인식 결과까지 받아서 `지금 javis가 무엇을 해야 하는지`를 결정하는 두뇌입니다.

## 역할 범위

### 엔진이 해야 하는 일

- 현재 상황 해석
- 다음 행동 결정
- 위험 신호 판단
- 상단장님에게 보여줄 짧은 설명 생성
- Codex에 보낼 보정/재지시 문구 생성

### 엔진이 하면 안 되는 일

- 창 포커스
- 클릭
- 붙여넣기
- 마우스 조작
- 안전정지 우회

즉 판단은 OpenAI가 하고, 실행은 로컬 런타임이 합니다.

## 1차 버전에서 다루는 판단 질문

v1에서 엔진이 답해야 하는 핵심 질문은 아래입니다.

### 1) 지금 계속 기다려야 하나요?

- 아직 Codex가 답변 중인지
- 화면 변화가 끝나지 않았는지
- 단계 완료 판단이 이른지

### 2) 다음 단계로 진행해도 되나요?

- 현재 단계 산출이 최소 기준을 충족했는지
- 위험 신호가 없는지
- 보류해야 할 이유가 없는지

### 3) 문제가 있다면 어떤 종류인가요?

- 일시 정지
- Codex 막힘
- 결과물 이상 가능성
- 판단 확신 부족
- 사람 확인 필요

### 4) 다음 액션은 무엇인가요?

- 계속 대기
- 다음 단계 진행
- 같은 단계 보정 프롬프트 전송
- 자동 진행 보류
- 상단장님 호출

## 판단 루프

`javis`의 판단 루프는 아래 순서로 동작합니다.

1. 로컬 런타임이 현재 상태를 수집
2. 프로젝트/단계/정책/화면 관련 컨텍스트를 정리
3. OpenAI 판단 엔진에 구조화 입력 전달
4. 엔진이 구조화 응답 반환
5. 로컬 런타임이 응답을 검증
6. 허용 가능한 행동만 실행
7. 결과를 상단장님에게 짧게 보고

## 입력 계층

판단 엔진에 들어가는 입력은 아래 5개 그룹으로 나눕니다.

### 1) Project Context

- 프로젝트 요약
- 목표 수준
- 전체 단계 목록
- 현재 단계 인덱스
- 현재 단계 목표

### 2) Runtime Context

- 자동 루프 상태
- 직전 액션
- 최근 실패 횟수
- 마지막 전송 시각
- 현재 대기 시간

### 3) Policy Context

- 운영 마스터 프롬프트
- 안전 규칙
- 단계 진행 판단 규칙
- 사용자 보고 톤 정책

### 4) Evidence Context

v1에서는 텍스트/메타데이터 중심, 이후 화면 인식이 추가됩니다.

- 최근 캡처 경로
- 안정화 감지 결과
- 로그 요약
- 최근 Codex 응답 요약
- 나중에는 비전/OCR 요약

### 5) User Override Context

- 상단장님 수동 승인 여부
- 상단장님 최근 피드백
- 특정 상황 예외 규칙

## 판단 출력 원칙

출력은 반드시 구조화되어야 합니다.

자유 텍스트만 받으면 생기는 문제:

- 자동 실행이 불안정함
- 예외 처리가 어려움
- 로그 분석이 어려움
- UI에 바로 표시하기 어려움

따라서 엔진은 아래 3개를 항상 줘야 합니다.

- `decision`
- `reason`
- `message_to_user`

그리고 가능하면 아래도 같이 줘야 합니다.

- `confidence`
- `risk_level`
- `next_prompt_to_codex`
- `needs_user_confirmation`

## 기본 결정 집합

v1에서는 아래 결정만 허용합니다.

- `continue`
  - 다음 단계 진행
- `wait`
  - 조금 더 관찰
- `retry`
  - 같은 단계 재지시 또는 보정
- `pause`
  - 자동 진행 멈춤
- `ask_user`
  - 상단장님 확인 요청

## 안전 원칙

### 1) 확신이 낮으면 공격적으로 움직이지 않음

- 낮은 확신에서는 `continue`보다 `wait` 또는 `ask_user`

### 2) 고위험이면 자동 진행 금지

- `risk_level = high`면 자동 진행 불가

### 3) Codex에 보내는 문구는 필요할 때만 생성

- 모든 턴에 새 프롬프트를 만들지 않음
- `retry` 또는 특정 `continue` 상황에서만 생성

### 4) 로컬 검증이 마지막 방어선

- 잘못된 구조 응답
- 허용되지 않은 decision
- 필수 필드 누락

이런 경우 로컬 런타임이 자동으로 `pause` 처리해야 합니다.

## Release별 엔진 진화

### Release 1

- 정책 편집 자리 확보
- 판단 엔진을 붙일 자리 확보
- 구조화 계약 방향 정리

### Release 2

- Codex-first 운영 문맥 정리
- prompt / 정책 / 운영 프로필 바닥 정리

### Release 3

- no automation / thread / project 운영 흐름 확정
- runboard / triage bridge 같은 운영 문맥 확보

### Release 4

- 실제 structured judgment 연결
- continue / wait / retry / pause / ask_user 분기
- confidence / risk 기반 안전 가드

### Release 5

- 화면 인식은 `항상 기본`이 아니라 `필요할 때만 켜는 visual evidence`로 사용
- 브라우저 결과 판독과 claim-vs-screen mismatch 기반 판단 강화

### Release 6

- 음성 질의응답과 동일 판단 엔진 공유

## 구현시 중요한 결론

1. OpenAI는 나중에 붙는 부가기능이 아니라 초반부터 들어가는 핵심 두뇌입니다.
2. 하지만 실행 권한은 끝까지 로컬 런타임이 잡고 있어야 합니다.
3. 입력은 점점 풍부해지되, 출력 계약은 처음부터 엄격해야 합니다.
4. 화면 인식과 음성은 결국 같은 판단 엔진을 공유하게 됩니다.
