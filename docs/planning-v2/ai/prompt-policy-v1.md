# Prompt Policy v1

## 문서 목적

이 문서는 상단장님이 `Control Center`에서 길게 설정하게 될 프롬프트를 어떤 구조로 나누고 관리해야 하는지 정의합니다.

핵심 원칙은 하나입니다.

`한 개의 거대한 프롬프트보다, 역할이 분리된 정책 모듈 여러 개가 더 안전하고 유지보수가 쉽다.`

## 왜 모듈형 프롬프트가 필요한가

긴 프롬프트 하나만 쓰면 아래 문제가 생깁니다.

- 어떤 규칙이 어떤 판단에 영향을 주는지 추적하기 어려움
- 작은 수정이 전체 판단 성향을 흔듦
- 화면 인식이 추가될 때 프롬프트가 비대해짐
- 음성까지 붙으면 더 엉킴

그래서 `javis`는 프롬프트를 계층으로 나눕니다.

## 프롬프트 계층

### 1) 운영 마스터 프롬프트

가장 상위 정책입니다.

- javis의 역할
- 상단장님 선호
- 기본 운영 철학
- 품질 우선순위
- 보수적/공격적 성향

예시 역할:

- Codex를 끝까지 운영하는 판단형 데스크톱 어시스턴트
- 확신이 낮으면 멈추고 설명
- 잘못된 진행보다 보수적 대기가 낫다

### 2) 단계 진행 판단 프롬프트

현재 단계가 끝났는지, 다음 단계로 가도 되는지 판단하는 정책입니다.

- 단계 완료 기준
- 다음 단계 전송 허용 조건
- 대기 조건
- 재시도 조건
- 사람 확인 필요 조건

### 3) 화면 판독 프롬프트

화면 인식이 붙었을 때 쓰는 정책입니다.

- Codex 화면에서 읽어야 하는 것
- 브라우저 화면에서 읽어야 하는 것
- 오류/이상/불완전 징후
- 품질 판단 기준

### 4) 재지시 생성 프롬프트

문제가 있을 때 Codex에 어떤 톤과 구조로 다시 지시할지 정하는 정책입니다.

- 짧고 명확한 수정 지시
- 원인 설명 우선
- 한 번에 한 수정 요청
- 필요 시 검증 요구 포함

### 5) 사용자 보고 프롬프트

상단장님께 보여줄 말투와 요약 형식을 정합니다.

- 짧은 존댓말
- 결론 우선
- 위험 시 명확하게
- 기술 디테일은 필요한 만큼만

### 6) 안전 규칙 프롬프트

아무리 다른 정책이 있어도 덮어쓰는 최상위 안전 규칙입니다.

- 애매하면 멈춤
- 반복 실패 시 자동 보류
- 고위험이면 사용자 호출
- 확신 낮으면 자동 진행 금지

## 실제 조합 순서

판단 시점마다 프롬프트를 아래 순서로 조합합니다.

1. 시스템 고정 규칙
2. 운영 마스터 프롬프트
3. 안전 규칙 프롬프트
4. 상황별 정책 프롬프트
5. 현재 단계/현재 증거 입력

즉 상단장님이 길게 적어둘 수는 있지만, 런타임에서는 무조건 역할별로 끼워 넣어야 합니다.

## Control Center에서의 편집 방식

상단장님이 실제로 편집하는 UI는 아래처럼 가는 것이 좋습니다.

- 탭 1: 운영 원칙
- 탭 2: 단계 진행 기준
- 탭 3: 화면 판독 기준
- 탭 4: 재지시 정책
- 탭 5: 사용자 보고 톤
- 탭 6: 안전 규칙

각 탭에는 아래가 있으면 좋습니다.

- 정책 설명
- 현재 입력 텍스트
- 기본 템플릿 복원
- 최근 수정 시각

## 기본 정책 예시

### 운영 원칙 예시

- 한 번에 안정적으로 끝낼 수 있는 범위만 진행
- 품질 확신이 낮으면 다음 단계로 넘기지 않음
- 작업 완료 시 검증 여부를 확인

### 단계 진행 기준 예시

- 산출이 존재하는지
- 명백한 실패가 없는지
- 추가 수정이 더 필요해 보이지 않는지

### 안전 규칙 예시

- 같은 단계 재시도 2회 초과 시 사람 호출
- 위험 신호가 보이면 자동 진행 중지
- 판단 확신 0.6 미만이면 자동 continue 금지

## 프롬프트 관리 원칙

### 1) 템플릿과 사용자 커스텀을 분리

- 기본 정책 템플릿
- 상단장님 커스텀 정책

둘을 분리 저장해야 합니다.

### 2) 버전 이력 보관

- 언제 어떤 정책이 바뀌었는지 남겨야 합니다.
- 판단 결과가 바뀐 이유를 추적할 수 있어야 합니다.

### 3) 지나치게 장문이면 경고

- 프롬프트가 너무 길어지면 유지보수가 어려워집니다.
- Control Center에서 과도한 길이를 경고해 주는 것이 좋습니다.

### 4) 정책 충돌 감지

- “확신 낮아도 진행”과 “확신 낮으면 중지” 같은 충돌은 감지해야 합니다.

## v1에서 필요한 최소 저장 구조

- `progress_policy`
- `vision_policy`
- `repair_policy`
- `report_policy`
- `safety_policy`

## 한 줄 결론

`javis`의 프롬프트는 하나의 초장문 텍스트가 아니라, 상황별로 조립되는 정책 시스템이어야 합니다.
