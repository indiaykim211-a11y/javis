# Sprint 2: Control Center and Resume

## 스프린트 목표

기존 콘솔 기능을 Control Center로 분리하고, 프로젝트/세션 복원 흐름을 제품 중심으로 정리합니다.

## 왜 이 단계가 필요한가

팝업만 예뻐도 설정과 운영이 여전히 복잡하면 제품 전체 인상은 바뀌지 않습니다. Sprint 2는 “보이는 화면”뿐 아니라 “들어가서 조정하는 화면”도 제품답게 만드는 단계입니다.

## 포함 티켓

- `R1-004` Control Center 창 분리
- `R1-005` 프로젝트/세션 홈
- `R1-006` 정책 편집기 스켈레톤

## 상세 목표

### 1) Control Center 역할 고정

- 기존 고급 제어를 별도 창으로 이동
- 탭 구조 기반 내비게이션 확정

### 2) 이어 쓰는 흐름 강화

- 최근 프로젝트 불러오기
- 현재 단계/마지막 저장/최근 판단 요약
- 팝업과 세션 홈 간 흐름 정리

### 3) 정책 편집 자리 확보

- 운영 정책
- 단계 판단 정책
- 안전 규칙

이런 텍스트를 실제로 다룰 수 있는 편집 틀 도입

## 완료 조건

- 메인 팝업과 Control Center 역할이 분리되어 보임
- 최근 프로젝트 복원 진입이 자연스러움
- 정책 편집 탭이 Release 2 연결을 위한 준비 상태를 갖춤

## 데모 시나리오

1. 팝업에서 `설정` 진입
2. Control Center에서 프로젝트 탭 확인
3. 최근 프로젝트 불러오기
4. 정책 탭에서 기본 프롬프트 확인/저장
5. 팝업으로 복귀

## 주요 리스크

- 기존 콘솔 기능이 그대로 노출되어 정보 밀도가 다시 높아질 수 있음
- 세션 복원과 프로젝트 전환이 섞이면 UX가 헷갈릴 수 있음
- 정책 편집기를 너무 일찍 과하게 만들면 Release 1 범위가 불필요하게 커질 수 있음
