# Release 4 Smoke Checklist v1

## 목적

`Release 4: Judgment Beta` 핵심 흐름을 5~10분 안에 반복 점검합니다.

핵심은 아래 5가지입니다.

- judgment input이 현재 프로젝트 기준으로 잘 묶이는가
- 판단 응답이 계약 기반으로 보이는가
- popup과 Control Center에 판단 결과가 잘 보이는가
- retry / ask_user / pause가 흐름으로 이어지는가
- low confidence / high risk가 과감한 continue로 새지 않는가

## 준비

1. 최신 Phase 3 운영 데이터가 있는 프로젝트를 엽니다.
2. `Codex 전략`과 `프로젝트` 카드가 최신인지 확인합니다.
3. 판단 엔진 테스트용 더미 결과 또는 실제 Codex 결과 1개를 준비합니다.

## 시나리오 A: continue 판단

1. 현재 단계 목표와 최근 결과 요약을 넣습니다.
2. judgment packet이 생성되는지 확인합니다.
3. 구조화 응답에서 `continue`를 시뮬레이션하거나 실제로 받습니다.

확인할 것:

- 현재 프로젝트 / 운영 모드 / 정책 / 최근 결과가 packet에 보입니다.
- popup에 `진행 가능` 성격 판단이 보입니다.
- 다음 행동이 명확합니다.

## 시나리오 B: retry 판단

1. 목표는 일부 충족했지만 미완 요소가 남은 결과를 준비합니다.
2. 구조화 응답에서 `retry`를 시뮬레이션하거나 실제로 받습니다.

확인할 것:

- retry 사유가 짧게 보입니다.
- Codex에 보낼 재지시 문구가 준비됩니다.
- 같은 단계 보정 흐름으로 이어질 수 있습니다.

## 시나리오 C: ask_user / pause 판단

1. 확신이 낮거나 위험이 있는 결과를 준비합니다.
2. 구조화 응답에서 `ask_user` 또는 `pause`를 시뮬레이션합니다.

확인할 것:

- 상단장님 확인 요청 문구가 분명합니다.
- 자동 continue가 막힙니다.
- 위험 또는 확신 부족 근거가 같이 보입니다.

## 시나리오 D: 잘못된 응답 강등

1. decision 누락
2. high risk continue
3. retry인데 next_prompt_to_codex 없음

같은 잘못된 응답을 시뮬레이션합니다.

확인할 것:

- validator가 응답을 그대로 통과시키지 않습니다.
- pause 또는 ask_user 쪽으로 안전하게 강등됩니다.

## 시나리오 E: 판단 이력

1. continue
2. retry
3. ask_user

같은 판단 2~3개를 연속으로 남깁니다.

확인할 것:

- 최근 판단과 이유가 timeline 형태로 보입니다.
- 왜 이런 결론이 났는지 다시 읽을 수 있습니다.

## 통과 기준

- 판단 입력과 출력이 계약 기반으로 정리됩니다.
- 상단장님이 현재 판단과 다음 행동을 바로 이해할 수 있습니다.
- 위험하거나 애매한 상황에서 제품이 보수적으로 동작합니다.
