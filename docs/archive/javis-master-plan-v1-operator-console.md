# javis 마스터플랜

## 1. 제품 비전

javis는 상단장님의 Codex 운영 역할을 대신하는 한국어 중심 데스크톱 오퍼레이터입니다.

궁극적인 목표는 아래와 같습니다.

- Codex 데스크톱 앱을 안정적으로 관찰하고 제어한다.
- Codex가 끝낸 작업을 화면 기준으로 이해한다.
- 결과물을 기준에 맞게 검수하고 다음 단계를 스스로 진행한다.
- 필요할 때만 사람을 호출하고, 평소에는 대기한다.
- 나중에는 웨이크워드, 음성대화, 음성 브리핑까지 지원한다.

한 줄 정의:

`Codex를 실무 단위로 운영하는 한국어 네이티브 AI 데스크톱 매니저`

## 2. 최종 상태 정의

javis의 최종 완성형은 아래처럼 동작해야 합니다.

1. 상단장님이 목표를 설명하면 javis가 이를 작업 계획, 품질 기준, 리스크 기준으로 구조화합니다.
2. javis가 Codex에 단계별 실행을 지시합니다.
3. Codex 작업이 끝날 때마다 javis가 화면, 로그, 브라우저 결과를 읽고 합격 여부를 판단합니다.
4. 합격이면 다음 단계로 넘어가고, 불합격이면 보정 프롬프트를 자동 전송합니다.
5. 상단장님은 중간 개입 없이 요약 리포트만 받거나, 음성으로 개입합니다.
6. 나중에는 “javis” 웨이크워드로 깨우고, 음성으로 “다음 단계 진행해”, “지금 상태 요약해”, “멈춰” 같은 명령을 내립니다.

## 3. 제품 원칙

### 3-1. 핵심 원칙

- 화면 기반으로 판단할 수 있어야 합니다.
- 좌표 클릭만 믿지 않고 의미 기반 제어로 진화해야 합니다.
- 자동 진행보다 안전 중단이 우선입니다.
- 모든 행동은 이유, 증거, 로그가 남아야 합니다.
- 한국어 사용성이 기본이어야 합니다.
- Codex 전용으로 시작하되 나중에 범용 운영자로 확장 가능해야 합니다.

### 3-2. 사용자 경험 원칙

- 상단장님이 “감독자”처럼 느껴져야 합니다.
- 버튼 수보다 흐름 명확성이 중요합니다.
- 앱은 복잡한 기능을 많이 담더라도 화면은 차분하고 빠르게 이해되어야 합니다.
- 긴 설명 대신 상태, 신뢰도, 다음 행동을 먼저 보여줘야 합니다.
- 음성 기능이 들어가도 시각적 통제권은 항상 남아 있어야 합니다.

## 4. 큰 단계 로드맵

javis 개발은 아래 8단계로 나누는 것이 좋습니다.

1. `Foundation`
2. `Codex Operator MVP`
3. `Screen Intelligence`
4. `Autonomous QA & Recovery`
5. `Project Brain`
6. `Korean Native UX`
7. `Voice javis`
8. `Platform & Scale`

아래부터 단계별로 하위 목표와 상세사항을 붙입니다.

## 5. 단계별 상세 계획

## Phase 1. Foundation

### 목표

개발 속도보다 구조를 먼저 잡아서, 이후 기능이 늘어나도 깨지지 않는 기반을 만드는 단계입니다.

### 하위 목표

- 앱 아키텍처 확정
- 런타임 모듈 분리
- 로그/설정/세션 저장 구조 확정
- 자동화 안정성 원칙 수립
- 기본 브랜딩과 디자인 시스템 방향 설정

### 상세 구현 항목

#### 1) 모듈 구조 정리

- `ui`
  - 데스크톱 인터페이스
- `automation`
  - 윈도우 포커스, 클릭, 캡처, 키 입력
- `perception`
  - OCR, 비전 판독, 상태 추출
- `orchestrator`
  - 단계 진행, 검수, 재시도, 중단 정책
- `memory`
  - 세션 기록, 프로젝트 상태, 판정 근거 저장
- `voice`
  - 웨이크워드, ASR, TTS
- `integrations`
  - Codex, 브라우저, 향후 외부 툴 연결

#### 2) 공통 데이터 모델 정리

- 프로젝트 정의
- 목표 수준 정의
- 단계 정의
- 액션 기록
- 화면 관측 결과
- 판정 결과
- 리커버리 결과
- 음성 대화 기록

#### 3) 상태 저장 체계

- 로컬 SQLite + 파일 스토리지 조합 추천
- 저장 대상
  - 프로젝트 메타데이터
  - 단계별 상태
  - 스크린샷
  - OCR 결과
  - 판정 사유
  - 보정 프롬프트
  - 사용자 개입 이력

#### 4) 안정성 기본 규칙

- 클릭 전 대상 창 확인
- 전송 전 프롬프트 프리뷰 저장
- 위험 행동은 재확인 또는 안전 모드
- 자동 루프는 항상 일시정지 가능
- 실패 시 연속 반복 횟수 제한

#### 5) 디자인 방향

- 톤: 조용한 프리미엄 운영 콘솔
- 언어: 한국어 기본, 전문적인 존댓말
- 느낌: 한국 S+급 B2C/B2B SaaS와 데스크톱 툴의 중간
- 시각 요소:
  - 밀도 높은 정보 설계
  - 또렷한 상태 색상 체계
  - “진행 가능 / 검토 필요 / 위험” 3단 상태
  - 다크/라이트 테마 모두 대응

### 완료 기준

- 앱 구조가 흔들리지 않음
- 설정과 세션이 안정적으로 저장됨
- 기본 디자인 토큰과 컴포넌트 규칙이 정해짐

## Phase 2. Codex Operator MVP

### 목표

상단장님의 현재 수동 역할을 가장 먼저 대체하는 단계입니다.

### 하위 목표

- Codex 창 탐지
- 안정적인 입력과 전송
- 단계별 자동 진행
- 사람 개입용 제어판 제공

### 상세 구현 항목

#### 1) Codex 창 제어

- 창 목록 탐색
- Codex 창 식별
- 포커스 복구
- 창 크기/위치 읽기
- 상대 좌표 기반 입력창 클릭
- 텍스트 붙여넣기 및 전송

#### 2) 운영 세션 구성

- 프로젝트 요약 입력
- 목표 수준 입력
- 단계 목록 입력
- 운영 규칙 입력
- 현재 단계 포인터 관리

#### 3) 기본 자동 루프

- 화면 캡처
- 캡처 간 유사도 비교
- 일정 시간 이상 안정 시 “완료 후보”로 판단
- 다음 단계 전송
- 전송 로그 기록

#### 4) 운영 UI

- 현재 단계
- 마지막 캡처 시간
- 최근 액션
- 다음 프롬프트 미리보기
- 자동 진행 on/off
- 긴급 정지
- 수동 검토 모드

### 완료 기준

- 상단장님이 손으로 “다음 단계 진행”을 누르던 역할을 최소 60~70% 대체
- 수동 확인 후 다음 단계 전송이 빠르고 안정적임

---

## Phase 3. Screen Intelligence

### 목표

javis가 화면을 “본다” 수준으로 끌어올리는 단계입니다. 이 단계부터 진짜 javis다운 느낌이 시작됩니다.

### 하위 목표

- Codex 화면 읽기
- 브라우저 결과 읽기
- 화면 요소 의미 파악
- 상태 추론 정확도 향상

### 상세 구현 항목

#### 1) 인식 파이프라인

- 스크린샷 수집
- OCR
- 영역 분할
- UI 요소 분류
- 텍스트 + 레이아웃 결합 해석

#### 2) Codex 상태 판독

판별해야 할 핵심 상태:

- 답변 생성 중
- 작업 완료 대기
- 에러 발생
- 추가 정보 요청
- 명령 입력 가능
- 긴 출력 후 정지

#### 3) 브라우저/결과물 판독

- 브라우저 창 탐지
- 현재 URL 추정 또는 제목 추적
- 페이지 로드 여부
- 콘솔 에러 수집 가능성 검토
- 스크린샷 기반 UI 품질 점검

#### 4) 시맨틱 액션 시스템

좌표 클릭 중심에서 아래 방식으로 진화:

- “다음 단계 버튼 위치 추정”
- “입력창 영역 감지”
- “오류 메시지 영역 탐지”
- “브라우저 미리보기 핵심 구역 탐지”

#### 5) 판독 신뢰도

모든 인식 결과에 아래를 붙여야 합니다.

- 신뢰도 점수
- 근거 스크린샷
- 근거 텍스트
- 실패 가능성 메모

### 완료 기준

- javis가 단순 캡처 비교가 아니라 화면 의미를 읽고 다음 행동 후보를 제시
- Codex와 브라우저 화면 상태를 합쳐 판단 가능

## Phase 4. Autonomous QA & Recovery

### 목표

Codex가 작업을 끝냈다고 해서 바로 다음으로 넘기지 않고, javis가 결과물을 검수하고 보정까지 수행하는 단계입니다.

### 하위 목표

- 자동 합격/보류/실패 판정
- 재시도 프롬프트 자동 생성
- 실패 원인 분류
- 검수 리포트 자동 생성

### 상세 구현 항목

#### 1) QA 기준 엔진

프로젝트마다 아래 기준을 정의 가능해야 합니다.

- 기능 완성도
- UI 품질
- 콘솔 오류 여부
- 반응형 여부
- 핵심 플로우 정상 여부
- 목표 수준 충족 여부

#### 2) 결과 검수 방식

- 스크린샷 비교
- OCR 텍스트 확인
- 브라우저 상태 확인
- 필요 시 테스트 명령 실행
- 이전 단계 요구사항과 현재 결과 매칭

#### 3) 실패 유형 분류

- 기능 미구현
- UI 깨짐
- 런타임 오류
- 요구사항 오해
- 임시방편 구현
- 설명은 그럴듯하지만 결과 미달

#### 4) 자동 리커버리

예시:

- “현재 결과는 모바일 레이아웃이 무너집니다. 이 문제만 수정해 주세요.”
- “기능은 동작하지만 목표 수준의 세련됨이 부족합니다. 히어로 섹션 품질을 높여 주세요.”
- “오류 메시지가 보입니다. 원인 분석 후 수정부터 진행해 주세요.”

#### 5) 승인 정책

- 자동 승인
- 자동 보정
- 사람 호출
- 즉시 중단

### 완료 기준

- 상단장님 개입 없이도 여러 단계를 이어서 진행 가능
- 실패 시 무한 반복이 아니라 품질 중심으로 보정

## Phase 5. Project Brain

### 목표

javis가 단순 매크로가 아니라 프로젝트의 문맥을 기억하는 운영 두뇌가 되도록 만드는 단계입니다.

### 하위 목표

- 프로젝트 메모리
- 목표-단계-산출물 연결
- 장기 작업 히스토리
- 의사결정 근거 축적

### 상세 구현 항목

#### 1) 프로젝트 메모리 계층

- 영구 메모리
  - 프로젝트 개요
  - 선호 스타일
  - 품질 기준
- 세션 메모리
  - 오늘 진행 상황
  - 현재 막힌 지점
  - 최근 보정 이력
- 액션 메모리
  - 어떤 프롬프트를 보냈는지
  - 어떤 결과가 나왔는지
  - 왜 다음 판단을 했는지

#### 2) 목표 추적

- 최종 목표를 세부 목표로 분해
- 단계별 충족도 계산
- 남은 위험 요소 추정
- 예상 완료 거리 추정

#### 3) 회고와 학습

- 어떤 프롬프트가 잘 먹히는지 기록
- 반복 실패 패턴 축적
- 상단장님이 자주 수정하는 포인트 학습

### 완료 기준

- 같은 프로젝트를 며칠 뒤 다시 열어도 맥락이 이어짐
- javis가 단순 진행기계가 아니라 실무 맥락을 설명 가능

---

## Phase 6. Korean Native UX

### 목표

한국 사용자 기준으로 진짜 “잘 만든 서비스”처럼 느껴지게 만드는 단계입니다.

### 하위 목표

- 한국어 중심 인터페이스 완성
- 고급 운영 대시보드
- 신뢰감 있는 카피라이팅
- 상단장님 취향에 맞는 관리자 경험

### 상세 구현 항목

#### 1) UX 구조

핵심 화면:

- 홈 대시보드
- 현재 프로젝트 화면
- Codex 세션 모니터
- QA 리포트 뷰
- 액션 로그 타임라인
- 설정
- 음성 모드 패널

#### 2) 홈 대시보드

보여줄 정보:

- 진행 중 프로젝트
- 현재 단계
- 최근 성공/실패 횟수
- 자동 진행 상태
- 사람 개입이 필요한 건수

#### 3) 프로젝트 화면

보여줄 정보:

- 목표 요약
- 단계 트리
- 현재 상태
- 위험 포인트
- 최근 산출물
- 리커버리 제안

#### 4) 시각 언어

- 전문적인 한국어 카피
- 부담스럽지 않은 프리미엄 느낌
- 밀도 높은 패널 구조
- 중요 상태 강조
- 스크린샷과 텍스트 리포트의 균형

#### 5) 편의 기능

- 단축키
- 최근 프로젝트 빠른 재개
- 자동 저장
- 단계별 메모
- 즐겨찾는 운영 규칙 템플릿

### 완료 기준

- 단순 개발자 툴이 아니라 제품다운 완성도
- 처음 보는 사람도 역할과 상태를 빠르게 이해

## Phase 7. Voice javis

### 목표

javis라는 이름에 맞는 음성 인터페이스를 완성하는 단계입니다.

### 하위 목표

- 웨이크워드
- 음성 명령
- 음성 브리핑
- 음성 대화 기반 운영

### 상세 구현 항목

#### 1) 웨이크워드

예시:

- “javis”
- “javis, 상태 알려줘”
- “javis, 다음 단계 진행해”

필요 요소:

- 저전력 대기
- 오탐 최소화
- 한국어 발화 최적화

#### 2) 음성 인식

지원 명령 예시:

- 다음 단계 진행해
- 지금 어디까지 됐어
- 왜 멈췄어
- 이 프로젝트 요약해
- 지금 보류하고 내일 이어서 하자
- 브라우저 결과 읽어줘

#### 3) 음성 응답

응답 스타일:

- 짧은 상태 요약
- 위험 시 즉시 경고
- 장문 설명은 요청 시에만
- 고급스럽고 과하지 않은 한국어 톤

#### 4) 대화형 운영 모드

예시:

- 상단장님: “javis, 다음 단계 전에 화면 괜찮은지 봐.”
- javis: “현재 모바일 레이아웃이 불안정해 보입니다. 수정 지시 후 진행할까요?”

### 완료 기준

- 키보드 없이 주요 명령을 운영 가능
- 음성 요약만으로도 현재 상태 파악 가능

## 6. 기술 스택 제안

## 6-1. 앱 셸

권장 순서:

1. 현재는 `Python + Tkinter`로 빠른 검증
2. 중기부터 `Tauri + React` 또는 `Electron + React`

판단:

- MVP는 Python이 빠릅니다.
- 고급 UX, 오디오, 패키징, 성능, 대시보드는 Tauri/Electron이 유리합니다.
- 장기적으로는 `Rust/Tauri + React 프론트 + Python worker` 혼합도 좋습니다.

## 6-2. 자동화 레이어

- Win32 API
- UI Automation
- 스크린 캡처
- SendInput 계열
- 브라우저 자동화 필요 시 Playwright 또는 유사 레이어

## 6-3. 인식 레이어

- OCR 엔진
- 비전 모델 연동
- UI 레이아웃 분석기
- 상태 판독 규칙 엔진

- 웨이크워드 엔진
- STT

- 마이크/오디오 장치 제어

## 6-5. 데이터

- 로컬 파일 스토리지

## 10. 상단장님 기준 추천 개발 순서

현실적으로 가장 좋은 순서는 아래입니다.

1. `Operator MVP`를 먼저 실전 수준으로 안정화
2. `Screen Intelligence`를 붙여 Codex/브라우저 읽기
3. `Autonomous QA & Recovery`로 자동 검수와 보정 완성
4. `Project Brain`으로 장기 문맥 기억
5. `Korean Native UX`로 제품 완성도 상승
6. 마지막에 `Voice javis`

이 순서가 좋은 이유:

- 상단장님의 현재 시간을 가장 빨리 절약합니다.
- 진짜 어려운 핵심인 “읽고 판단하는 능력”에 빨리 들어갈 수 있습니다.
- 음성은 임팩트가 크지만, 운영 두뇌가 먼저 단단해야 합니다.

## 12. 한 줄 결론

javis는 충분히 가능합니다.

다만 핵심은 “클릭 자동화 앱”이 아니라, `Codex를 읽고 판단하고 다시 지시하는 한국어 운영 두뇌`로 설계하는 것입니다.

그 관점으로 가면, 지금 만든 뼈대는 좋은 출발점이고 다음 승부처는 분명합니다.

`화면 인식 -> 상태 판독 -> 품질 검수 -> 자동 보정 -> 음성 운영`
