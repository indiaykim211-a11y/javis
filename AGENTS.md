인코딩/한글 파일 작업 주의사항

이 프로젝트는 한글 텍스트가 많으므로 인코딩 손상(모지바케)을 매우 주의할 것.

규칙:
- 한글이 포함된 파일(.ts, .tsx, .js, .jsx, .json, .md, .py, .env, .txt 등)은 shell 기반 벌크 편집을 최대한 피할 것.
- 특히 PowerShell의 Get-Content, Set-Content, Out-File, 리다이렉션(>, >>), 파이프라인을 이용한 “읽고 다시 쓰기” 방식으로 한글 파일 전체를 재저장하지 말 것.
- 가능하면 apply_patch 중심으로 수정하고, 수정 범위는 가능한 한 작게 유지할 것.
- 작은 범위만 정확히 수정하고, 파일 전체를 통째로 다시 쓰는 방식은 피할 것.
- 파일 이동/이름 변경이 필요하면 내용 재기록 없이 move/rename/git mv 방식으로 처리할 것.
- 정말 PowerShell로 파일을 다시 써야 하면 인코딩을 명시적으로 지정하고, 기존 파일의 인코딩(BOM/no-BOM)을 먼저 확인한 뒤 동일하게 유지할 것.
- repo의 텍스트 파일 인코딩은 UTF-8로 가정하고, BOM/no-BOM 정책은 기존 파일 상태를 함부로 바꾸지 말 것.
- PowerShell 5.x 환경에서는 UTF-8 no-BOM 한글 파일이 깨질 수 있으므로 특히 더 조심할 것. 가능하면 PowerShell 7 계열 + UTF-8 일관성을 유지할 것.
- 한글이 많은 파일은 shell 명령으로 대량 치환, 대량 포맷, 대량 재저장하지 말 것.
- 편집 후 diff에서 한글이 깨진 흔적(예: ???, 諛⑹궛, �, 의미 없는 한자/특수문자 조합)이 보이면 즉시 작업을 중단하고 더 수정하지 말고 사용자에게 알릴 것.
- 인코딩이 의심되면 먼저 수정하지 말고 현재 파일 인코딩 상태와 안전한 수정 방법부터 설명할 것.
- 대규모 리팩토링이나 파일 분리 작업은 한 번에 크게 하지 말고, 먼저 작은 단위로 수정 후 결과를 확인할 것.
